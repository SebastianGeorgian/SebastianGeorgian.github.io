Greenhouse Monitor

bachelor's thesis · runs 24/7

A smart greenhouse that watches itself: six sensor types wired to a Raspberry Pi 5 feed a PostgreSQL database; a secured Flask dashboard renders live charts and heatmaps; a local TFT screen shows conditions at a glance — and when flame or gas is detected, the buzzer fires and a Telegram alert reaches my phone within seconds.

Field & edge

Six sensor types cover the conditions that matter in a greenhouse: DHT11 for temperature and humidity, soil-moisture, rain, flame and MQ-6 gas — climate on one side, safety on the other. Everything is wired to a Raspberry Pi 5 over GPIO.

Raw pins are noisy, so inputs are sampled at 10 Hz and debounced before a value is trusted. Climate readings are averaged over 6-minute windows on the device — the database stores signal, not noise. Safety sensors skip the averaging: flame or gas trips the local buzzer (SOS pattern) immediately.

Storage

Validated readings are inserted into PostgreSQL as time-stamped rows in a sensor_readings schema built for the questions the dashboard actually asks — trends, ranges and extremes over time, not single points. Access goes through connection pooling, so the dashboard, the display and the alert engine can read concurrently without exhausting the database.

Dashboard & alerts

The main window into the system is a Flask dashboard with live charts and heatmaps, behind authentication and rate limiting, served by gunicorn. Next to the greenhouse itself, a small TFT screen (ST7735, SPI) shows current conditions at a glance — no phone or laptop needed.

A separate alert engine polls for danger conditions every 5 seconds: if flame or gas is detected, a Telegram message lands on my phone within seconds — the difference between finding out now and finding out tomorrow.

Reliability

A monitoring system is only useful if it is running when something goes wrong. Every component runs as a systemd service with auto-restart — if a process dies, it comes back on its own. The system runs 24/7 without anyone babysitting it, which is the entire point.

What I'd build next

  • MQTT between sensor nodes and the backend, so devices publish to a broker instead of being wired into the application — adding a sensor shouldn't mean touching the backend.
  • TimescaleDB on top of PostgreSQL once reading volume outgrows the plain relational schema.
  • LoRa for greenhouses out of Wi-Fi range — long distance, low power, small payloads: exactly the shape of this data.
  • A bare-metal microcontroller node in C (ESP32 / STM32) reading a sensor over I2C and reporting over UART — moving the acquisition end from Linux down to the metal.

mygit

mygit — Git, from scratch

systems programming · C11 · no libraries

A working subset of Git, written in C11 with zero external dependencies — including the SHA-256 hash function, implemented from scratch and verified against the official NIST vectors. It doesn't shell out to the real Git or fake the behaviour with file copies: it uses the same core model — a content-addressable object store of blobs, trees and commits, a binary staging area and refs — so checkout genuinely reconstructs any snapshot from hashes alone.

Content addressing

The idea the whole thing rests on: an object's name is the SHA-256 of its contents. Not a name I choose — the content dictates its own address. Every object is serialised as "<type> <size>\0<payload>", hashed over the whole thing (header included, so a blob and a tree holding the same bytes can't collide), and stored at objects/<first 2 hex>/<rest> — the 2-character split keeps directories small enough that the filesystem stays fast.

Three things fall out of that for free: deduplication (two identical files — or a thousand — are stored once, and the code for it is a single if), immutability (an object can't change without changing its name, so history can't be quietly rewritten), and verifiability (re-hash any object and compare it to its own filename — bit rot shows up immediately).

Binary serialization

The index is a custom binary format: a magic number (MGIX, so a wrong file is rejected instead of misread), a version (so the format can evolve without corrupting old repos), then length-prefixed entries — all integers big-endian.

Written out byte by byte, deliberately: dumping the struct straight to disk would hard-code the compiler's padding and the CPU's byte order into the file, so a repo written on x86 could read back wrong on ARM. Commits, by contrast, are plain text — different file, different criteria: the index is rewritten constantly and never read by a human, a commit is written once and read during debugging.

Building trees from a flat index

The index is a flat sorted list; a commit needs a hierarchy of tree objects, one per directory. The observation that solves it: because the list is sorted with strcmp, every path inside a given directory is contiguous — no foreign path can sort between two paths sharing a prefix.

So one linear scan per level groups the children, and each group is recursed into first — subtrees are written before their parents, and the parent embeds the child's hash as data. No hash map, no extra sorting, no auxiliary structures. The case that catches naive implementations: dir.txt sorts before dir/x (ASCII: '.' < '/'), and isn't inside dir at all — so matching the prefix isn't enough, the separator has to follow. There's a test for exactly that.

Correctness & safety

14 end-to-end scenarios, including a 4 KB binary file from /dev/urandom compared byte-for-byte after a full add → commit → checkout cycle, nested directories, staged deletions, detached HEAD via abbreviated hash, and running from a subdirectory. CI runs all of it on gcc and clang with -Werror, plus the suite under AddressSanitizer and UBSan — which found a real bug: qsort called with a NULL pointer on an empty vector. It ran fine, but it's undefined behaviour.

The parts that destroy data are the ones written most carefully: every file is written to a temp file and renamed atomically, so a crash can't leave a half-written index behind; and checkout refuses to run at all if it would overwrite uncommitted work or an untracked file — checked before anything is touched, not halfway through.

Where it goes next

  • diff — the real one: Myers' algorithm, which is dynamic programming over the edit graph. The biggest remaining piece.
  • zlib compression for objects, which are currently stored raw — the project's first external dependency, and a clear space/CPU trade-off.
  • merge — commits with two parents, finding the common ancestor in the DAG, three-way conflict detection. What turns a mini-Git into a Git.
  • .mygitignore and cat-file — small, and the natural next commits.
